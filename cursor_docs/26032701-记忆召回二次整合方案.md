# 记忆召回二次整合方案（向量 + 图）

本文面向你当前脚本 `GD25/express_customer_service copy.py` 的实际问题：  
在 `retrieve_context()` 中只使用了 `mem0.search()` 返回的 `results`（向量记忆），未对 `relations`（图记忆）进行二次整合，导致：

- 图检索已经执行并产生耗时，但没有转化为上下文价值；
- 提示词上下文只包含“事实片段”，缺少“关系路径/约束关系/实体连边”信息；
- 长期看会出现“检索成本上升但回复质量未同步提升”。

---

## 1. 当前链路与问题定位

你当前单轮对话链路为：

1. `retrieve_context()` 调用 `self.mem0.search(query, user_id=user_id)`
2. 仅读取 `memories["results"]` 拼接为 `serialized_memories`
3. 将该文本写入系统上下文并生成回复
4. 调用 `self.mem0.add(...)` 写回记忆

而在 mem0 内部（`mem0/memory/main.py`）：

- `search()` 会并发执行：
  - `_search_vector_store(...)`
  - `graph.search(...)`（启用 graph_store 时）
- 最终返回：
  - `results`：向量召回结果
  - `relations`：图召回结果

因此，当前核心漏洞不是“没检索到图数据”，而是“检索到了但未融合使用”。

---

## 2. mem0 视角：推荐的整合思路

### 2.1 利用 mem0 原生双通道返回（最低改造成本）

`mem0.search()` 已经给出双路数据：

- `results`: 语义相似记忆（适合补充用户偏好、事实）
- `relations`: 实体关系（适合做一致性、约束、因果关联）

可直接在应用层增加“二次融合器（Fusion Layer）”：

1. 归一化向量条目与关系条目（统一字段：`source`, `evidence`, `score`, `type`）
2. 去重（文本归一、实体对归一）
3. 重排序（按置信度 + 时效性 + 覆盖度）
4. 预算裁剪（token budget）
5. 结构化注入提示词（事实区 + 关系区）

### 2.2 用图结果做“约束校验”而非只做补充

对于客服场景，图结果建议承担“校验层”角色：

- 当向量事实与图关系冲突时，以更新更晚、置信度更高者优先；
- 对关键字段（姓名/地址/电话/会员等级）做一致性检查；
- 仅将通过校验的结论注入最终上下文，避免错误放大。

### 2.3 限制 graph 检索开销

若图结果未使用，不应默认每次都跑完整图检索。可以采用：

- 路由触发：仅当 query 命中“身份、关系、历史轨迹、核对”意图时启图；
- 动态阈值：图检索阈值、limit 随意图变化；
- 超时降级：图检索超时则仅返回向量结果并记录指标。

## 3. LangChain / LangGraph 可借鉴方案

虽然你当前直接用 mem0 + LangChain ChatModel，但可借鉴成熟的“检索编排”思路。

### 3.1 LangChain：Multi-Retriever + Contextual Compression

常见组合：

- Retriever A：向量检索（事实）
- Retriever B：图查询（关系）
- 合并器：RRF/MMR/加权排序
- 压缩器：Contextual Compression（减少 token）

可迁移到你当前工程的做法：

- 把 `results` 和 `relations` 分别包装为 Document 列表；
- 统一进入压缩和重排模块；
- 最后按“事实摘要 + 关系摘要 + 风险提示”注入 prompt。

### 3.2 LangGraph：状态机化检索流程（推荐中期演进）

LangGraph 常见图：

1. Query Intent Node（识别是否需要图）
2. Vector Retrieve Node
3. Graph Retrieve Node（可并行）
4. Fusion Node（冲突消解 + 去重 + 预算裁剪）
5. Answer Node
6. Memory Write Node（异步优先）

优势：

- 每个节点可独立打点，定位瓶颈更清晰；
- 可做条件分支（低价值 query 不走图）；
- 易于加入重试、超时、熔断策略。

---

## 4. 其他成熟解决方案对比（可选参考）

### 4.1 GraphRAG 思路（知识图谱优先）

适用于关系密集问答，但成本较高。  
在你的客服场景中，更适合“轻量 GraphRAG”：

- 向量负责 recall；
- 图负责 consistency / path evidence；
- 最终仍由一个回答模型整合。

### 4.2 Hybrid RAG（向量 + BM25 + 图）

业界常用“三路召回 + 统一重排”：

- 向量：语义相似
- BM25：关键词精确命中（如运单号、手机号）
- 图：实体关系链路

客服场景收益点：

- 对结构化标识符问题（单号、地址关键词）召回更稳；
- 减少“语义像但业务不相关”的误召回。

### 4.3 Two-Stage Retrieval（粗召回 + 精排）

第一阶段：高召回（vector+graph）  
第二阶段：交叉编码器或 LLM rerank（保留 top-k）

适合你后续规模变大时使用，当前可先做轻量版打分融合。

## 5. 针对你当前工程的可执行方案（分阶段）

## Phase 1：最小可用改造（建议先做）

目标：不改 mem0 内核，仅改业务层 `retrieve_context()`。

改造点：

1. 从 `mem0.search()` 同时读取 `results` 与 `relations`
2. 构建两个上下文块：
  - `历史事实`（来自 results）
  - `关系证据`（来自 relations，格式如 `A -[R]-> B`）
3. 增加简单融合规则：
  - 同实体重复条目去重
  - 事实条目上限 N，关系条目上限 M
  - 超长时优先保留高置信和关键字段
4. 结构化喂给 prompt，而不是拼成一行长文本

收益：

- 立刻让“图检索耗时”转换成“模型可用证据”；
- 不动底层，风险最低。

## Phase 2：性能优化（建议紧随其后）

1. 为 `retrieve_context` 打点：
  - `mem0.search.total_ms`
  - `vector_ms`
  - `graph_ms`
  - `fusion_ms`
2. 设定图检索触发策略（意图路由）
3. 设定图检索超时与降级
4. 记忆写入改为“异步落库”（先返回用户，再后台 add）

## Phase 3：编排升级（中期）

将检索与融合迁移到 LangGraph：

- 状态显式化；
- 分支可控；
- 可观测性和稳定性更好。

## 6. 融合设计建议（可直接落到代码）

建议新增一个统一结构：

- `MemoryEvidence`
  - `kind`: `fact` | `relation`
  - `text`: 展示文本
  - `score`: 置信分
  - `source`: `vector` | `graph`
  - `entity_keys`: 实体集合（用于去重）
  - `timestamp`: 可选（用于新鲜度）

融合打分建议（示例）：

- `final_score = 0.5 * semantic_score + 0.3 * recency_score + 0.2 * structure_score`
- 对关键字段（姓名/地址/电话）冲突项加惩罚；
- 对能形成闭环证据链（事实 + 关系互证）的条目加分。

---

## 7. 上下文注入模板建议

建议从“单段拼接”改成“分区注入”，例如：

- 区块1：用户稳定事实（高置信）
- 区块2：关系证据（图谱）
- 区块3：不确定信息与待确认项

并在系统提示词中明确：

- 若事实与关系冲突，优先输出“待用户确认”；
- 对隐私字段只做必要回显；
- 回答后可附“我已依据以下信息核对：...”。

---

## 8. 风险与注意事项

1. 图数据质量风险：实体抽取错误会污染关系边，需要定期清洗与阈值调优；
2. 成本风险：图写入路径 LLM 调用较多，需控制触发频率；
3. 时延风险：若无超时降级，单轮尾延迟会持续恶化；
4. 冲突风险：多来源事实不一致时需显式冲突策略，不能简单覆盖。
