## mem0 `add` 调用链路与关键技术说明（ExpressCustomerService 场景）

### 1. 入口：客服侧保存一轮对话

在 `GD25/express_customer_service copy.py` 中，一轮对话结束后会调用：

- **调用点**：`ExpressCustomerService.save_interaction`
- **核心代码**：

```python
interaction = [
    {"role": "user", "content": user_input},
    {"role": "assistant", "content": assistant_response},
]
self.mem0.add(interaction, user_id=user_id)
```

**要点**：
- `interaction` 是一个 **List[Dict]**，每个元素包含 `role` 与 `content`。
- 通过 `user_id` 将这一轮对话与某个用户的长期记忆空间绑定。
- 这里不传 `agent_id`、`run_id`、`memory_type`，因此会走 **“普通用户记忆”** 分支，而不是程序化记忆。

### 2. Mem0 外层接口：`Memory.add` 的输入规范

在 `mem0/memory/main.py` 中，`add` 方法的签名为：

```python
def add(
    self,
    messages,
    *,
    user_id: Optional[str] = None,
    agent_id: Optional[str] = None,
    run_id: Optional[str] = None,
    metadata: Optional[Dict[str, Any]] = None,
    infer: bool = True,
    memory_type: Optional[str] = None,
    prompt: Optional[str] = None,
):
    ...
```

**关键点**：
- **会话维度**：`user_id` / `agent_id` / `run_id` 三者中必须有一个，用于确定记忆归属的“会话空间”。
- **消息格式容错**：
  - `str` → 自动包装成 `[{"role": "user", "content": messages}]`
  - `dict` → 包装为单元素 list
  - 其他类型 → 抛出 `Mem0ValidationError`
- **记忆类型**：
  - 当 `memory_type` 为 `"procedural_memory"` 且传入 `agent_id` 时，走 `_create_procedural_memory` 分支。
  - 当前客服场景未传 `memory_type`，因此创建的是 **通用对话记忆（语义 + 情节）**。

### 3. 元数据与过滤条件构建：`_build_filters_and_metadata`

调用链中的第一步是：

```python
processed_metadata, effective_filters = _build_filters_and_metadata(
    user_id=user_id,
    agent_id=agent_id,
    run_id=run_id,
    input_metadata=metadata,
)
```

**设计目的**：
- 将 `user_id` / `agent_id` / `run_id` 等标识与用户传入的业务字段统一合并为 `processed_metadata`。
- 同时生成 `effective_filters`，用于 **向量库检索与图数据库查询的过滤条件**。

**效果**：
- 后续所有存储与检索，都可以通过这些过滤条件，在多用户 / 多 Agent / 多 Session 情况下做到 **严格隔离与高效查询**。

### 4. 消息解析与视觉信息支持：`parse_vision_messages`

在确认不是程序化记忆后，代码会根据配置决定是否启用视觉消息解析：

```python
if self.config.llm.config.get("enable_vision"):
    messages = parse_vision_messages(messages, self.llm, self.config.llm.config.get("vision_details"))
else:
    messages = parse_vision_messages(messages)
```

**关键技术点**：
- 统一处理包含图片/多模态内容的消息，将其转换为 LLM 可理解的结构化输入。
- 当启用视觉能力时，会把图片信息通过模型能力抽取为文字描述/特征，便于后续事实抽取与向量化。

### 5. 并行写入：向量库与图存储双通路

`add` 方法在核心存储阶段使用了 `ThreadPoolExecutor` 做并行处理：

```python
with concurrent.futures.ThreadPoolExecutor() as executor:
    future1 = executor.submit(self._add_to_vector_store, messages, processed_metadata, effective_filters, infer)
    future2 = executor.submit(self._add_to_graph, messages, effective_filters)

    concurrent.futures.wait([future1, future2])

    vector_store_result = future1.result()
    graph_result = future2.result()
```

**技术设计要点**：
- **双存储通路**：
  - `_add_to_vector_store`：负责生成向量并写入向量数据库，承载 **语义相似度检索**。
  - `_add_to_graph`：负责构建实体/关系图谱，支持更复杂的 **知识图谱推理和关系查询**。
- **并行化优势**：
  - 两条通路互不依赖，使用线程池并行可以在 I/O 与模型推理较重的场景中显著减少总耗时。
  - 对于客服这种高 QPS 场景，能有效提高吞吐量。

**返回结果**：
- 如果启用了 `enable_graph`：
  - 返回 `{"results": vector_store_result, "relations": graph_result}`
- 否则：
  - 返回 `{"results": vector_store_result}`

### 6. 向量化与记忆抽取：`_add_to_vector_store`

在 `_add_to_vector_store` 中，根据 `infer` 的值走两条不同路径：

#### 6.1 `infer=False`：直接向量化原始消息

场景：希望 **不做事实抽取**，而是把每条消息当作独立记忆写入。

核心逻辑：
- 遍历所有 `messages`，过滤掉无效/`system` 消息。
- 对每条消息：
  - 复制元数据 `metadata`，补充 `role`（以及可选的 `actor_id`）。
  - 调用 `embedding_model.embed(msg_content, "add")` 生成向量。
  - 调用 `_create_memory` 写入存储，并返回 `{"id", "memory", "event", "actor_id", "role"}`。

#### 6.2 `infer=True`（默认）：LLM 驱动的事实抽取与去重更新

当前客服场景走的就是这一条路径。

关键步骤：

1. **消息预处理**：`parsed_messages = parse_messages(messages)`
   - 将多轮对话结构化为事实抽取友好的文本格式，便于大模型阅读理解。

2. **系统 Prompt 生成**：
   - 如果配置了 `custom_fact_extraction_prompt`，则使用自定义提示词。
   - 否则，根据是否为 Agent 记忆选择不同模板：

   ```python
   is_agent_memory = self._should_use_agent_memory_extraction(messages, metadata)
   system_prompt, user_prompt = get_fact_retrieval_messages(parsed_messages, is_agent_memory)
   ```

   - `_should_use_agent_memory_extraction` 的逻辑：
     - `metadata` 中存在 `agent_id`
     - 且 `messages` 中包含 `assistant` 角色
     - 满足上述条件则认为是 **Agent 记忆**，使用专用提示模板。

3. **调用 LLM 抽取事实**：

   ```python
   response = self.llm.generate_response(
       messages=[
           {"role": "system", "content": system_prompt},
           {"role": "user", "content": user_prompt},
       ],
       response_format={"type": "json_object"},
   )
   ```

   - 要求返回 JSON 对象，通常包含 `"facts"` 字段。
   - 使用 `remove_code_blocks` 清洗可能带有 Markdown 代码块包装的返回内容。
   - 再通过 `json.loads` 解析出事实列表。

4. **事实级别的去重 / 更新策略**：
   - 对每条新抽取的 fact，会：
     - 生成嵌入向量。
     - 与已有记忆进行相似度匹配，判断是“新增事实”还是“更新已有事实”。
     - 最终形成 `"event": "ADD" / "UPDATE"` 等结果。

**整体效果**：
- 从“整轮对话”中自动抽取出 **稳定的、可复用的知识/偏好/属性**，避免把大量闲聊噪音直接写入向量库。
- 支持记忆的 **增量更新**（例如：地址变更、联系方式变更），而不是简单堆叠。

### 7. 图存储通路：`_add_to_graph`（概要）

虽然在阅读片段中没有完整展示 `_add_to_graph` 实现，但从接口设计可以推断：

- 使用 `messages` 与 `effective_filters` 构建图结构：
  - 抽取实体（例如：用户、快递单号、地址、时间节点等）。
  - 抽取实体之间的关系（例如：`用户-下单->运单`、`运单-收货地址->地点`）。
- 将这些实体与关系写入图数据库或图结构存储中，为后续：
  - 复杂路径查询（谁在什么时间在哪个地址收过快递）
  - 规则型推理/推荐
提供支持。

### 8. 结合客服场景的整体链路总结

以一轮具体对话为例：

1. **对话结束**：
   - 用户输入 `user_input`，模型生成 `assistant_response`。
2. **调用保存接口**：
   - `ExpressCustomerService.save_interaction` 构造两条消息并调用：
     - `self.mem0.add(interaction, user_id=user_id)`
3. **Mem0 处理**：
   - 统一构建 `metadata` 与 `filters`，确保记忆与该 `user_id` 绑定。
   - 解析多轮消息，判断是否是 Agent 记忆（当前不是）。
   - 进入 `infer=True` 的事实抽取路径：
     - 用 LLM 从整轮对话中提炼出关键事实（如：收件地址、快递偏好、问题类型等）。
     - 对事实做去重/更新，避免冗余。
4. **并行存储**：
   - 向量库中写入这些事实的向量表示，支持后续 **相似语义检索**。
   - 图存储中写入实体与关系，支持 **结构化查询与关系推理**。
5. **后续检索**：
   - 在下一轮对话中，`retrieve_context` 会通过 `mem0.search(query, user_id=user_id)`:
     - 利用同一套过滤条件与向量搜索，从用户历史记忆中找出相关事实。
     - 以自然语言串联为“相关历史信息”注入到 Prompt 中，提高回复的一致性与个性化。

### 9. 对你理解代码的建议关注点

- **接口层**：`ExpressCustomerService.save_interaction` 只关心“本轮输入+输出”，不关心内部实现，这是一个典型的 **高内聚、低耦合** 设计。
- **记忆抽象层**：`Memory.add` 把多种输入形式统一为标准消息列表，同时通过 `user_id/agent_id/run_id` 与 `memory_type` 做不同记忆空间与类型的路由。
- **智能处理层**：利用 LLM 做事实抽取与去重更新，使得“记忆”是 **结构化且可维护的知识**，而不是简单日志堆积。
- **存储层**：并行写入向量存储与图存储，分别承担“模糊语义检索”和“精确结构推理”的职责。

掌握以上几个层次，就能比较完整地理解从客服脚本调用 `self.mem0.add(...)` 到底在 mem0 内部发生了什么，以及它为何能在后续对话中提供“有记忆感”的上下文支持。
