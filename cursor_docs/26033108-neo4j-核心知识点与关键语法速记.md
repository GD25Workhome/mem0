# 1. 10 条核心结论（先记这个）

1. **先建模，再写查询**：查询决定模型，不是反过来。  
2. **节点放实体，关系放语义**：关系方向要统一，关系属性承载“交互事实”。  
3. **唯一约束是生产底线**：没有唯一约束，`MERGE` 也可能写出重复实体。  
4. **写入默认用 `MERGE`（幂等）**，补充 `ON CREATE` / `ON MATCH` 分支。  
5. **删除节点优先 `DETACH DELETE`**，避免“节点仍有关联关系”报错。  
6. **批量写入用 `UNWIND`**，不要逐条单发。  
7. **多步查询靠 `WITH` 管道化**，同时做变量裁剪和聚合分段。  
8. **路径查询是图数据库价值核心**：可变长路径 + 最短路径。  
9. **性能优化先看 `EXPLAIN/PROFILE`**：关注是否走索引、DB Hits、是否路径爆炸。  
10. **数据会“长脏”**：要定期巡检孤立点、重复点、异常高出度、异常属性值。

# 2. 建模速记（Schema 设计）

## 2.1 三步法

1. 列业务问题（要回答什么查询）。  
2. 定节点标签与唯一键（`uid` / `pid` / `oid`）。  
3. 定关系类型、方向、关系属性（如 `since`、`score`、`created_at`）。

## 2.2 建模判断口诀

- **频繁被单独查询的对象** -> 节点  
- **描述两个对象之间一次交互** -> 关系（关系上放属性）  
- **时间、评分、权重等“连接事实”** -> 优先放关系属性

## 2.3 常见反模式

- 关系类型全叫 `RELATED`（语义丢失）。  
- 同一关系方向不统一（查询复杂、易错）。  
- 无唯一键直接上 `MERGE`（并发下重复数据风险高）。

# 3. 关键语法模板区（高频可直接改）

## 3.1 约束与索引

```cypher
// 唯一约束：幂等写入前置条件
CREATE CONSTRAINT user_uid_unique IF NOT EXISTS
FOR (u:User) REQUIRE u.uid IS UNIQUE;

// 存在约束（按版本能力选择）
CREATE CONSTRAINT user_uid_exists IF NOT EXISTS
FOR (u:User) REQUIRE u.uid IS NOT NULL;

// 普通索引：提升过滤/查询性能
CREATE INDEX user_name_idx IF NOT EXISTS
FOR (u:User) ON (u.name);
```

## 3.2 Create / Merge（写入）

```cypher
// 明确新建（可能重复）
CREATE (u:User {uid: 'u1', name: 'Alice'});

// 幂等写入（推荐默认写法）
MERGE (u:User {uid: $uid})
ON CREATE SET
  u.name = $name,
  u.created_at = timestamp()
ON MATCH SET
  u.name = $name,
  u.updated_at = timestamp();
```

```cypher
// 幂等建立关系（两端先匹配）
MATCH (a:User {uid: $from_uid}), (b:User {uid: $to_uid})
MERGE (a)-[r:KNOWS]->(b)
ON CREATE SET r.since = date($since);
```

## 3.3 Read（查询）

```cypher
// 基础过滤 + 排序 + 分页
MATCH (u:User)
WHERE u.city = $city
RETURN u.uid, u.name
ORDER BY u.updated_at DESC
SKIP $offset LIMIT $limit;
```

```cypher
// 可选关系：即使没有关系也返回主节点
MATCH (u:User {uid: $uid})
OPTIONAL MATCH (u)-[:PLACED]->(o:Order)
RETURN u, collect(o) AS orders;
```

## 3.4 Update（更新）

```cypher
MATCH (u:User {uid: $uid})
SET u.name = $new_name, u.updated_at = timestamp();
```

```cypher
// 删除属性 / 删除标签
MATCH (u:User {uid: $uid})
REMOVE u.nickname
REMOVE u:TempLabel;
```

## 3.5 Delete（删除）

```cypher
// 删关系
MATCH (:User {uid: $uid1})-[r:KNOWS]->(:User {uid: $uid2})
DELETE r;
```

```cypher
// 删节点及其全部关系
MATCH (u:User {uid: $uid})
DETACH DELETE u;
```

## 3.6 WITH（多阶段查询管道）

```cypher
MATCH (u:User)-[:PLACED]->(o:Order)
WITH u, count(o) AS order_cnt
WHERE order_cnt >= 3
RETURN u.uid, order_cnt
ORDER BY order_cnt DESC;
```

关键点：
- `WITH` 后未显式传递的变量会被丢弃。  
- 先 `WITH` 再 `WHERE`，常用于“聚合后过滤”。  

## 3.7 UNWIND（批量写入）

```cypher
UNWIND $rows AS row
MERGE (u:User {uid: row.uid})
SET u.name = row.name,
    u.updated_at = timestamp();
```

## 3.8 路径查询与最短路径

```cypher
// 1~3 跳可达
MATCH p = (a:User {uid:'u1'})-[:KNOWS*1..3]->(b:User)
RETURN b.uid, length(p) AS hops
ORDER BY hops, b.uid;
```

```cypher
// 最短路径（无向写法：-[]-）
MATCH p = shortestPath(
  (a:User {uid:'u1'})-[:KNOWS*..10]-(b:User {uid:'u9'})
)
RETURN p, length(p) AS hops;
```

# 4. 事务并发与幂等（生产最小原则）

1. 先建唯一约束，再开放写入。  
2. 接口写入统一 Upsert 风格：`MERGE + ON CREATE/ON MATCH`。  
3. 事务尽量短，减少大事务和热点冲突。  
4. 对可恢复错误做有限重试（指数退避）。  
5. 不把“重试”当幂等替代品：幂等由模型与约束保证。

# 5. 性能优化速查

## 5.1 先看什么

- `EXPLAIN`：先看计划，不执行。  
- `PROFILE`：看真实执行、DB Hits、是否索引命中。  

## 5.2 高频优化动作

- 过滤条件尽量绑定到有索引/约束字段。  
- 避免不相关 `MATCH` 并列导致笛卡尔积。  
- 路径查询限制跳数（如 `*1..3`），防止路径爆炸。  
- 大批处理使用 `UNWIND + 分批`。

# 6. 数据质量治理（巡检模板）

## 6.1 孤立节点

```cypher
MATCH (u:User)
WHERE NOT (u)--()
RETURN count(u) AS isolated_cnt;
```

## 6.2 重复实体（同业务键）

```cypher
MATCH (u:User)
WITH u.uid AS uid, count(*) AS cnt
WHERE uid IS NOT NULL AND cnt > 1
RETURN uid, cnt
ORDER BY cnt DESC;
```

## 6.3 异常高出度节点

```cypher
MATCH (n)-[r]->()
WITH n, count(r) AS out_degree
RETURN labels(n) AS labels, n, out_degree
ORDER BY out_degree DESC
LIMIT 20;
```
