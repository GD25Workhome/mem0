## Mem0 图记忆管理：零经验学习路径（基于本仓库实现）

你已经知道“图”的概念，但对图数据库/图工具链/业务落地没经验。下面给出一条**可一步步执行**的学习路线，目标是让你从“看得懂 `Memory._add_to_graph` / `graph.search`”走到“能用图做记忆管理与业务增强”，并且每一步都对齐本仓库的真实实现文件。

---

## 0. 先建立正确的心理模型：Mem0 的“图”在这里做什么

在本仓库里，Mem0 的图通路不是取代向量记忆，而是**并行的补充索引/补充检索**：

- **向量库**：以 “data(文本) + embedding(向量)” 为主，用来做语义相似检索（主路径）。
- **图存储**：把文本里的“实体—关系—实体”抽出来，写成图结构，用来做：
  - 结构化关系检索（谁和谁有什么关系）
  - 关系上下文补充（返回 `relations`，给业务做二次利用）

在 `mem0/memory/main.py` 中很直观：

- 写入：`add()` 会并发执行 `_add_to_vector_store(...)` 和 `_add_to_graph(...)`
- 查询：`search()` 会并发执行 `_search_vector_store(...)` 和 `graph.search(...)`（如果启用图）

因此你的学习目标应该分成两层：

- **第一层（能跑通/能看懂）**：图是如何被构建、如何被查询、如何被 user/agent/run 隔离。
- **第二层（能用起来）**：如何把 `relations` 真正用于“记忆管理/业务增强”（例如召回、解释、约束、对齐）。

---

## 1. 入口处先读懂：图通路在 Mem0 主流程里的位置

### 1.1 写入入口：`Memory._add_to_graph`

从你标注的代码可知，写入图时 Mem0 做了两件事：

- **过滤/隔离**：如果 filters 里没有 user_id，会填一个默认值（注意这是一种兜底行为）
- **数据拼接**：把非 system 的 message content 用换行拼成 `data`，交给 `self.graph.add(data, filters)`

学习要点：

- 你要关心的不是这段拼接，而是：`filters`（user/agent/run）如何决定“图里写到哪个命名空间/子图”。

### 1.2 查询入口：`Memory.search` 里的 `graph.search`

你标注的 `future_graph_entities = executor.submit(self.graph.search, query, effective_filters, limit)` 表明：

- 图查询与向量查询并发
- 图查询输入同样受 `effective_filters`（user/agent/run）约束
- 图查询结果会通过 `{"relations": ...}` 返回

学习要点：

- 先搞清楚 `relations` 的结构长什么样、业务如何消费它（哪怕你暂时不消费，也要知道“它提供了什么信息”）。

---

## 2. 先确认“你现在实际用的是哪一种图后端”

本仓库支持多个图 provider，通过工厂类切换：

- `mem0/utils/factory.py::GraphStoreFactory`
  - `memgraph`
  - `neptune` / `neptunedb`
  - `kuzu`
  - `default`（当前实现是 `mem0/memory/graph_memory.py`）

学习要点：

- 你想“学会如何用图管理记忆”，第一步是**确定你当前配置里 graph_store.provider 是什么**，因为不同 provider 的依赖、部署、查询语法细节会差别很大。

建议的最小阅读顺序：

1. `mem0/configs/base.py`（MemoryConfig 里 graph_store 怎么配置）
2. `mem0/graphs/configs.py`（GraphStoreConfig 支持哪些字段）
3. `mem0/utils/factory.py`（provider → 实现类 的映射）

你会得到一个明确答案：现在走的是 `graph_memory.py`（Neo4jGraph）、还是 `graphs/neptune/`*（Neptune）。

---

## 3. 从“存取 API”入手：把图当成一个小型子系统来理解

不管 provider 是什么，本仓库的图子系统对外最关键的 API 基本一致：

- `add(data, filters)`：把文本增量写入图（会产生新增/删除的关系变更）
- `search(query, filters, limit)`：返回与 query 相关的若干条三元组关系（source, relationship, destination）
- `delete_all(filters)`：按 user/agent/run 范围清空子图
- `get_all(filters, limit)`：拉取某范围内的关系概览

学习要点：

- 把 `filters` 当作“租户/命名空间”，把 `add/search/delete_all/get_all` 当作“图记忆 CRUD+检索”。

---

## 4. 深入理解：图是怎么从文本里“长出来”的（核心机制）

以默认实现 `mem0/memory/graph_memory.py::MemoryGraph` 为例（其他 provider 也遵循类似分解），“从文本到图”的过程可以拆成四段：

### 4.1 实体抽取（Entity Extraction）

- 输入：一段文本（data）
- 输出：实体列表 + 实体类型（entity_type_map）
- 手段：LLM + tool schema（`extract_entities` 工具）

你要掌握的不是 prompt 细节，而是：

- **实体命名规范**（例如统一 lower/下划线）
- **user_id 的自指处理**（“我/我的”映射到 filters['user_id']）

### 4.2 关系抽取（Relation Extraction）

- 输入：文本 + 实体列表
- 输出：关系列表（source, relationship, destination）
- 手段：LLM + tool schema（`relations` 工具）

关键学习点：

- relationship 会被 sanitize（避免 Cypher 不合法字符）
- 关系是 Mem0 图记忆的核心数据单位（可理解为“结构化记忆”）

### 4.3 节点/关系写入与“去重合并”（Graph Merge）

实现上会尝试：

- 用 embedding 在图里找“相近实体节点”（阈值来自 GraphStoreConfig.threshold）
- 找到就复用节点，找不到就创建新节点
- 用 MERGE 创建/复用关系，并维护 mentions/时间戳等属性（不同 provider 不同）

你要重点理解的概念：

- **threshold 的业务含义**：阈值越高，越不容易把相近实体合并到同一节点；越低，越容易合并但可能误合并。

### 4.4 删除策略（Graph Pruning）

默认实现里有一个“删除关系”的步骤：

- 先基于当前文本产生的实体，在图里搜索已有相关关系（search_output）
- 再让 LLM 决策哪些关系需要删除
- 执行删除

你要理解它的定位：

- 这是“记忆管理”层面的关键：图不是只增不减，而是允许结构化关系随时间更新/清理。

## 5. 查询侧理解：为什么 `graph.search` 不是“图数据库的全文检索”

`graph.search(query, filters, limit)` 的典型做法是：

1. 从 query 中抽取实体
2. 在图里找这些实体附近的关系（出边/入边）
3. 对候选关系做重排（默认实现使用 BM25 对三元组序列做 rerank）
4. 返回若干条三元组

学习要点：

- 图检索的输出是**关系三元组**，它更偏“可解释结构”，而不是“相似文本段落”。
- 所以图检索适合补充：
  - 约束（用户—偏好—xxx）
  - 事实链路（用户—地址—xxx—订单—xxx）
  - 可解释上下文（为什么推荐/为什么这样答）

## 6. 实操学习路线（建议你按顺序走，每一步都有“完成标志”）

### Step A：只做阅读，先把“图通路全貌”串起来（1～2 小时）

- **读主入口**：`mem0/memory/main.py`（`_add_to_graph`、`search` 并发图查询、`delete_all` 调用图的 delete_all）
- **读工厂映射**：`mem0/utils/factory.py::GraphStoreFactory`
- **读配置模型**：`mem0/graphs/configs.py::GraphStoreConfig`
- **读默认实现**：`mem0/memory/graph_memory.py`

完成标志：

- 你能用自己的话描述：开启图以后，add/search 返回结构如何变化（`relations` 从哪里来）。

### Step B：跑通一次最小闭环（半天，偏实践）

目标是验证你“理解的路径”是否正确。建议最小闭环包括：

- `add(messages, user_id=...)` 触发图写入
- `search(query, user_id=...)` 拿到 `relations`
- `get_all(filters)` 或 `delete_all(filters)` 做一次范围操作（看清隔离是否生效）

完成标志：

- 你能稳定复现：同一个 user_id 能看到关系；换一个 user_id 查不到（或查到很少）。

### Step C：学会“调参 + 控制质量”（1～2 天）

围绕三个最影响体验的点做实验：

- **GraphStoreConfig.threshold**：调高/调低，观察实体合并/拆分的变化
- **custom_prompt**：引导关系抽取更贴近业务（例如“只抽取用户偏好/收货信息/投诉原因”）
- **删除策略的风险**：观察删除是否会误删（理解它依赖 LLM 决策，属于不确定环节）

完成标志：

- 你能解释：为什么某些实体被合并/没合并，为什么某些关系会被删除。

### Step D：把图用于“记忆管理”的业务落地（持续迭代）

你可以从最简单的两种用法开始：

- **用图做“可解释记忆”**：把 `relations` 转成自然语言补充上下文，告诉模型“用户—偏好—xxx/用户—地址—xxx”。
- **用图做“结构化约束”**：当图中存在某些关系时，作为规则触发器（例如存在“用户—投诉—丢件”，则优先走安抚+流程模板）。

完成标志：

- 你能给出 1～2 个业务场景，明确说明“为什么向量检索不够、图能补什么”，并能用 Mem0 返回的 `relations` 落地。

---

## 7. 你接下来最该关注的 5 个“新人坑”

- **坑 1：没搞清 provider 就开始学**：Neo4j/Memgraph/Neptune/Kuzu 的依赖与行为差异很大，先确定当前配置走哪条实现。
- **坑 2：把图当成全文检索**：图返回的是结构三元组，价值在“关系/约束/解释”，不是替代向量库的语义召回。
- **坑 3：忽视 filters（user/agent/run）**：图也需要严格隔离；否则会出现跨用户关系混杂，后果比向量串库更难排查。
- **坑 4：threshold 没调就谈效果**：实体合并质量高度依赖阈值，默认值不一定适合你的业务实体分布。
- **坑 5：删除策略不做监控**：图的删除依赖 LLM 决策，建议你先在测试数据上观察误删概率，再考虑线上启用或加保护。
