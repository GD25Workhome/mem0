## 0. 练习目标与注意事项

### 目标
1. 理解“唯一约束”如何防重复数据。  
2. 理解“普通索引”如何加速过滤查询。  
3. 学会用 `EXPLAIN/PROFILE` 比较执行计划差异。

### 注意事项
- 本练习使用独立标签：`IndexDemoUser`，避免污染你现有 mem0 图。  
- 所有演示数据都带 `session: 'idx_practice_260331'`，便于最后一键清理。  
- 为了更清楚看到索引效果，建议至少造 **5000** 条数据（本文给了 10000 条版本）。

## 3. 造数（建议 10000 条）

> 说明：这里用 `UNWIND range(...)` 一次性造数据。  
> 数据中 `uid` 唯一，`name` 和 `city` 会重复，方便做索引过滤实验。

```cypher
UNWIND range(1, 10000) AS i
CREATE (u:IndexDemoUser {
  session: 'idx_practice_260331',
  uid: 'u_' + toString(i),
  name: 'name_' + toString(i % 200),   // 200 个重复名
  city: 'city_' + toString(i % 20),    // 20 个重复城市
  score: i % 100,
  created_at: timestamp()
});
```

验证数据量：

```cypher

RETURN count(u) AS total_users;
```

## 5. 创建“唯一约束”（对应练习要求 1）

> 唯一约束建议作用在业务主键上。  
> 这里用 `uid`，并且约束包含 `session`，避免未来跨 session 测试互相影响。

```cypher
CREATE CONSTRAINT index_demo_uid_unique IF NOT EXISTS
FOR (u:IndexDemoUser)
REQUIRE (u.session, u.uid) IS UNIQUE;
```

再次查看约束：

```cypher
SHOW CONSTRAINTS
YIELD name, type, labelsOrTypes, properties
WHERE name = 'index_demo_uid_unique'
RETURN *;
```

### 5.1 验证唯一约束真的生效

先创建一条用户：

```cypher
CREATE (:IndexDemoUser {session: 'idx_practice_260331', uid: 'u_unique_test', name: 'dup_test'});
```

再插入同 session + 同 uid（应失败，报唯一约束冲突）：

```cypher
CREATE (:IndexDemoUser {session: 'idx_practice_260331', uid: 'u_unique_test', name: 'dup_test_2'});
```

> 看到冲突报错是正确现象，说明“防重复”生效。

## 6. 创建“普通索引”（对应练习要求 2）

> 针对高频过滤字段建索引。  
> 这里给 `(session, name)` 建联合索引，服务于 `session + name` 查询。

```cypher
CREATE INDEX index_demo_session_name_idx IF NOT EXISTS
FOR (u:IndexDemoUser) ON (u.session, u.name);
```

确认索引 ONLINE：

```cypher
SHOW INDEXES
YIELD name, state, type, labelsOrTypes, properties
WHERE name = 'index_demo_session_name_idx'
RETURN *;
```

## 9. 常见问题与排查

### 问题 1：创建唯一约束时报重复冲突

原因：库里已有重复 `(session, uid)`。  
处理：
1. 先查重复键：
```cypher
MATCH (u:IndexDemoUser {session: 'idx_practice_260331'})
WITH u.session AS session, u.uid AS uid, count(*) AS cnt
WHERE cnt > 1
RETURN session, uid, cnt;
```
2. 清理重复后再建约束。

### 问题 2：索引建了但查询仍像全扫描

可能原因：
- 查询条件没有按索引字段写（例如只按 `name` 查，但索引是 `(session, name)`）
- 索引还未 ONLINE
- 数据量太小，优化器认为扫描更便宜
