# Neo4j 事务并发一致性与 MySQL 对比：生产实践与最小练习

对应总览章节：[`26033102-neo4j-关键应用场景与CRUD进阶学习清单.md`](26033102-neo4j-关键应用场景与CRUD进阶学习清单.md) 的 `## 7. 事务与并发一致性`。

---

## 1. 先给结论（生产中怎么做）

在生产里，Neo4j 和 MySQL 都会做同一件事：**把“唯一性约束”放在数据库层，把“重试和幂等”放在应用层**。

常见做法：
1. 数据库层强约束（唯一键/唯一约束）  
2. 应用层使用幂等写入（MySQL 常用 `INSERT ... ON DUPLICATE KEY UPDATE`，Neo4j 常用 `MERGE`）  
3. 遇到瞬时冲突或锁竞争，做有限重试（指数退避）  
4. 一次业务操作放在一个事务边界里提交

## 2. Neo4j vs MySQL：并发一致性的核心差异

### 2.1 共同点

- 都支持 ACID 事务。  
- 都可能在高并发下发生冲突（唯一键冲突、锁等待、死锁/瞬时失败）。  
- 都建议：**约束兜底 + 应用重试**。

### 2.2 Neo4j（图）侧特点

- 幂等写入常用 `MERGE`。  
- `MERGE` 本身不是“绝对去重魔法”，仍需要唯一约束兜底，否则高并发下可能出现重复实体。  
- 写入热点节点（很多事务同时改同一节点）时，容易出现竞争更新，需要拆分热点或批处理。

典型写法：

```cypher
MERGE (u:User {uid: $uid})
ON CREATE SET u.created_at = timestamp()
ON MATCH SET u.updated_at = timestamp()
RETURN u;
```

### 2.3 MySQL（关系）侧特点

- 幂等写入更常见为唯一索引 + upsert 语句。  
- 事务隔离级别和行锁行为更可预期（大多数团队经验丰富）。  
- 强事务链路（订单、支付）通常仍优先 MySQL。

典型写法（示意）：

```sql
INSERT INTO user(uid, name, created_at)
VALUES (?, ?, NOW())
ON DUPLICATE KEY UPDATE
  name = VALUES(name),
  updated_at = NOW();
```

## 3. 生产实践：大家怎么落地（可直接套用）

### 实践 1：先建唯一约束，再谈并发幂等

Neo4j：

```cypher
CREATE CONSTRAINT user_uid_unique IF NOT EXISTS
FOR (u:User) REQUIRE u.uid IS UNIQUE;
```

MySQL：
- `uid` 建 `UNIQUE KEY`

### 实践 2：写入接口必须幂等

- 同一个业务请求重复提交，最终结果只保留一份实体。  
- 对外暴露幂等键（如 `request_id`）是常见做法。

### 实践 3：有限重试（只重试可恢复错误）

- 重试次数：通常 2~5 次  
- 退避策略：100ms -> 200ms -> 400ms  
- 不重试业务逻辑错误（只重试锁冲突/瞬时网络错误）

### 实践 4：缩小事务范围

- 只把“必须原子”的步骤放在同一事务里  
- 大批量写入用分批事务，避免长事务拖垮吞吐

## 4. 最小练习案例（可在你的库中直接执行）

目标：验证“并发/重复写入同一实体，最终只保留 1 个节点”。

> 为了不影响现有数据，使用独立标签 `TxnDemoUser`，并用 `session` 隔离。

### 4.1 清理环境

```cypher
MATCH (n:TxnDemoUser {session:'txn_practice_260331'})
DETACH DELETE n;
```

### 4.2 建唯一约束（关键）

```cypher
CREATE CONSTRAINT txndemo_uid_unique IF NOT EXISTS
FOR (u:TxnDemoUser) REQUIRE (u.session, u.uid) IS UNIQUE;
```

### 4.3 准备“幂等写入”语句

把下面语句作为你的标准写入模板（之后在两个 Browser 窗口或快速重复执行中使用）：

```cypher
MERGE (u:TxnDemoUser {session:'txn_practice_260331', uid:'u_concurrent_1'})
ON CREATE SET
  u.name = 'first_create',
  u.created_at = timestamp(),
  u.hit_count = 1
ON MATCH SET
  u.hit_count = coalesce(u.hit_count, 0) + 1,
  u.updated_at = timestamp()
RETURN u;
```

### 4.4 模拟“并发/重复提交”

方法 A（手工近似并发）：
1. 打开两个 Neo4j Browser 标签页  
2. 几乎同时执行 4.3 的语句若干次

方法 B（顺序快速重复）：
- 连续执行 4.3 语句 5~10 次

### 4.5 验证结果（最关键）

```cypher
MATCH (u:TxnDemoUser {session:'txn_practice_260331', uid:'u_concurrent_1'})
RETURN count(u) AS node_count, u.uid AS uid, u.hit_count AS hit_count, u.created_at AS created_at, u.updated_at AS updated_at;
```

期望：
- `node_count = 1`（不会出现多份节点）  
- `hit_count` 随重复执行增长

## 5. 再做一个“反例实验”（帮助理解约束价值）

> 这一步在练习结束后可做，可选。

1. 删除约束：  
```cypher
DROP CONSTRAINT txndemo_uid_unique IF EXISTS;
```

2. 改用 `CREATE`（非幂等）连续执行多次：  
```cypher
CREATE (:TxnDemoUser {session:'txn_practice_260331', uid:'u_bad_case'});
```

3. 查询重复：  
```cypher
MATCH (u:TxnDemoUser {session:'txn_practice_260331', uid:'u_bad_case'})
RETURN count(u) AS duplicated_count;
```

你会看到：没有约束 + 非幂等写法，很容易重复。

---

## 6. 你应记住的生产原则（简版）

1. **唯一约束是底线，不是可选项。**  
2. **MERGE/Upsert 是默认写入姿势。**  
3. **重试只针对可恢复错误，且次数有限。**  
4. **事务尽量短，小批量分段提交。**
