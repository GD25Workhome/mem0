## Mem0 记忆检索、记忆新增与底层存储结构——架构级梳理（基于项目真实实现）

本文档面向“技术架构/实现机制”视角，结合业务侧调用场景（`GD25/express_customer_service copy.py`）与 Mem0 内部实现（`mem0/memory/main.py`、`mem0/vector_stores/pgvector.py`、`mem0/memory/storage.py`），梳理：

- **记忆查询（search）**：如何做会话隔离、向量检索、（可选）图检索与重排序，以及结果结构如何被业务侧消费。
- **记忆新增（add）**：如何把一轮对话转成“事实级记忆”，并对既有记忆做 ADD/UPDATE/DELETE（LLM 决策），以及写入向量库与历史库的落点。
- **底层存储结构**：向量库（以 PGVector 为例）的表结构、payload 组织、过滤条件在 SQL 中如何体现；以及 SQLite history 记录的角色与字段。

> 约束：所有结论均以当前仓库代码为依据；对“可能的风险/语义偏差”会明确标注为**实现观察**，不把推断当事实。

---

## 1. 业务侧使用场景：为什么需要 Mem0

以 `ExpressCustomerService` 为例，Mem0 在一轮对话中的职责非常清晰：**先查记忆 → 组装上下文 → 生成回复 → 再写记忆**。

### 1.1 查询入口：把“相关历史信息”注入 Prompt

业务侧调用点（`retrieve_context`）是：

- `memories = self.mem0.search(query, user_id=user_id)`
- 把 `memories["results"][i]["memory"]` 拼成一段文本，注入到 LLM 上下文中。

这一层的关键设计是：**业务只需要 `query + user_id`，不关心向量化/检索/过滤/排序的细节**；Mem0 负责把“记忆系统”封装成一个高内聚组件。

### 1.2 新增入口：把一轮对话写入 Mem0

业务侧调用点（`save_interaction`）是：

- 构造 `interaction = [{"role":"user",...},{"role":"assistant",...}]`
- `self.mem0.add(interaction, user_id=user_id)`

这里同样体现了抽象边界：业务只提供**对话原材料**与**会话范围（user_id）**，Mem0 内部决定“抽取哪些事实、如何去重/更新、如何存储”。

## 2. Mem0 的架构分层：从接口到存储的三层抽象

结合 `mem0/memory/main.py` 的实现，Mem0 的能力可以按职责分成三层：

1. **接入层（API/Facade）**：`Memory.add` / `Memory.search` 作为稳定接口，负责参数规范化、会话范围校验、并发编排、返回结构统一。
2. **记忆编排层（Extraction + Policy）**：
   - `add(infer=True)`：LLM 抽取事实 → 检索可能相关旧记忆 → LLM 决策 ADD/UPDATE/DELETE/NONE → 执行相应写入/更新/删除。
   - `search`：构建过滤条件 → 向量检索 →（可选）rerank → 返回规范化 MemoryItem。
3. **存储与索引层（Vector Store + History + Graph）**：
   - **向量库**：负责向量近邻检索与 payload 过滤（例如 PGVector 的 `payload->>key = value`）。
   - **history（SQLite）**：记录每条 memory 的变更事件与前后值（ADD/UPDATE/DELETE）。
   - **图存储（可选）**：`enable_graph` 时并行写入与查询 `relations`。

你可以用下图把整体数据流串起来（流程图为“同步 add/search”主路径）：

```mermaid
flowchart LR
  A[业务侧<br/>ExpressCustomerService] -->|search(query,user_id)| B[Memory.search]
  B --> C[_build_filters_and_metadata<br/>effective_filters]
  C --> D[_search_vector_store<br/>embed(query)+vector_store.search]
  D --> E[results: MemoryItem[]]
  B -->|可选 enable_graph| F[graph.search]
  B -->|可选 rerank| G[reranker.rerank]
  B --> H[返回 {"results":..., "relations"?:...}]

  A -->|add(messages,user_id)| I[Memory.add]
  I --> J[_build_filters_and_metadata<br/>metadata+filters]
  I -->|并发| K[_add_to_vector_store]
  I -->|并发 enable_graph| L[_add_to_graph]
  K --> M[vector_store.insert/update/delete]
  K --> N[SQLite history.add_history]
  I --> O[返回 {"results":..., "relations"?:...}]
```

## 3. 会话隔离：Mem0 如何确保“只在该用户范围内”读写

无论 `add` 还是 `search`，都会首先调用 `_build_filters_and_metadata(...)` 构造两类信息：

- **用于写入的 metadata 模板**：`base_metadata_template`（后续会被写入向量库 payload）
- **用于查询的过滤条件**：`effective_query_filters`（后续会被传给向量库的 search/list）

实现要点（`mem0/memory/main.py::_build_filters_and_metadata`）：

- `user_id` / `agent_id` / `run_id` **至少提供一个**，否则抛 `Mem0ValidationError(VALIDATION_001)`。
- 对于每个提供的会话标识：
  - 写入 `base_metadata_template["user_id"]=...`
  - 也写入 `effective_query_filters["user_id"]=...`
- 可选 `actor_id` 只用于查询过滤（写入 `effective_query_filters`），不会进入 `base_metadata_template`。

从架构角度看，这个函数相当于 Mem0 的“多租户隔离/命名空间”根机制：后续所有向量检索都以这些过滤条件为边界进行收敛。

## 4. 记忆查询（Memory.search）：从 query 到 results 的真实链路

### 4.1 `Memory.search` 的并发编排

在 `Memory.search(query, *, user_id/agent_id/run_id, limit, filters, threshold, rerank)` 中：

- 先通过 `_build_filters_and_metadata(..., input_filters=filters)` 生成 `effective_filters`。
- 校验 `effective_filters` 中至少包含 `user_id/agent_id/run_id` 之一（否则抛 `ValueError`）。
- 使用线程池并发执行：
  - `self._search_vector_store(query, effective_filters, limit, threshold)`
  - 若 `enable_graph=True`，并发 `self.graph.search(query, effective_filters, limit)`
- 若 `rerank=True` 且配置了 `self.reranker`，对向量检索结果做 rerank（失败会降级为原结果）。

### 4.2 `_search_vector_store`：向量化、检索与结果“扁平化”

关键步骤（`mem0/memory/main.py::_search_vector_store`）：

1. **向量化**：`embeddings = self.embedding_model.embed(query, "search")`
2. **向量检索**：`memories = self.vector_store.search(query=query, vectors=embeddings, limit=limit, filters=filters)`
3. **规范化输出**：把向量库返回的 `OutputData(id, score, payload)` 转为 `MemoryItem` 字典：
   - `memory = payload["data"]`
   - `hash/created_at/updated_at` 从 payload 取
   - `score = mem.score`（由向量库返回）
   - 会把 `user_id/agent_id/run_id/actor_id/role` 这些“promoted keys”提升到顶层字段
   - payload 中其余字段被归入 `metadata`（如果存在）
4. **threshold 过滤**：实现是 `if threshold is None or mem.score >= threshold: keep`

> 实现观察（不做推断）：在 PGVector 中 `score` 实际是 SQL 返回的 `distance`（L2 距离，越小越相似），但 `_search_vector_store` 的 threshold 过滤条件是 `>=`。这意味着 **threshold 的语义是否与“越大越相似”一致，取决于具体 vector store 的 score 定义**；在使用 PGVector 且启用 threshold 时，需要特别注意这一点。

### 4.3 返回结构：业务侧应依赖的稳定形态

`search` 的返回是：

- 无图：`{"results": original_memories}`
- 有图：`{"results": original_memories, "relations": graph_entities}`

业务侧（客服示例）只使用 `memories["results"][i]["memory"]`，因此只要 `MemoryItem.memory` 稳定即可完成“上下文注入”。

## 5. 记忆新增（Memory.add）：从对话到“事实级记忆”的真实链路

### 5.1 输入规范化与分支选择

`Memory.add(messages, *, user_id/agent_id/run_id, metadata, infer=True, memory_type, prompt)` 的关键点：

- **messages 容错**：
  - `str` → 包装成 `[{"role":"user","content":...}]`
  - `dict` → 包装成单元素 list
  - 其他类型 → `Mem0ValidationError(VALIDATION_003)`
- **程序化记忆分支**：
  - 当 `agent_id is not None` 且 `memory_type == "procedural_memory"`，走 `_create_procedural_memory(...)`
  - 否则走“通用对话/事实记忆”流程（客服场景就是这条）

### 5.2 并行写入：向量库与图存储

`add` 使用线程池并发：

- `self._add_to_vector_store(messages, processed_metadata, effective_filters, infer)`
- `self._add_to_graph(messages, effective_filters)`（仅 enable_graph 时真正写入）

返回结构同 search：是否带 `relations` 取决于 `enable_graph`。

### 5.3 `_add_to_vector_store` 两条路径：`infer=False` 与 `infer=True`

#### 5.3.1 `infer=False`：把每条消息原样写入

实现逻辑是逐条消息：

- 跳过 `role == "system"` 的消息
- 为每条消息复制一份 metadata，并写入：
  - `role`
  - （可选）`actor_id`：来自消息的 `name`
- 对 `content` 向量化：`embed(msg_content, "add")`
- `_create_memory` 写入向量库，并在返回中标记 event=ADD

适用特征：更接近“对话日志向量化”，不做事实抽取与更新策略。

#### 5.3.2 `infer=True`（默认）：LLM 抽取事实 + LLM 决策记忆动作

这是客服场景的默认路径，核心链路分 4 步：

**A. LLM 抽取事实（facts）**

- 先把多轮消息规整：`parsed_messages = parse_messages(messages)`
- 选择抽取 prompt：
  - 若配置 `custom_fact_extraction_prompt` 则用它
  - 否则根据 `_should_use_agent_memory_extraction(messages, metadata)` 决定模板（agent memory / user memory）
  - 最终调用 `get_fact_retrieval_messages(parsed_messages, is_agent_memory)` 得到 system/user prompt
- 调用 LLM：`generate_response(..., response_format={"type":"json_object"})`
- 清洗 + JSON 解析，得到 `new_retrieved_facts`（列表）

**B. 为每条新 fact 检索“可能相关”的旧记忆**

对每条 `new_mem in new_retrieved_facts`：

- 对该 fact 计算 embedding（并缓存到 `new_message_embeddings`）
- 用该 embedding 在向量库中 search（limit=5），过滤条件只取 `user_id/agent_id/run_id`（命名为 `search_filters`）
- 把命中的旧记忆收集成 `retrieved_old_memory = [{"id": mem.id, "text": payload["data"]}, ...]`
- 用 dict 去重（按 `id`）

这一步在架构上非常关键：Mem0 并不是把“所有旧记忆”喂给 LLM，而是先用向量检索做一次**候选收敛**，再让 LLM 在“小集合”上做更新策略。

**C. LLM 决策：对每条新 fact 应当 ADD / UPDATE / DELETE / NONE**

为避免 LLM“胡编 UUID”，实现做了一个防护：

- 把候选旧记忆的 UUID 映射成 `"0","1","2"...` 这种临时 id（`temp_uuid_mapping`）
- 把被替换后的候选集与新 facts 一起，通过 `get_update_memory_messages(...)` 生成“动作决策提示词”
- 让 LLM 返回一个 JSON，其中 `memory` 是动作列表，每项包含：
  - `event`：ADD / UPDATE / DELETE / NONE
  - `text`：目标文本
  - `id`：若是 UPDATE/DELETE/NONE，需要引用候选旧记忆的临时 id

**D. 执行动作：落到向量库与 history**

逐条动作执行：

- **ADD**：`_create_memory(action_text, existing_embeddings=new_message_embeddings, metadata=deepcopy(metadata))`
- **UPDATE**：`_update_memory(memory_id=映射回真实 UUID, data=action_text, ...)`
- **DELETE**：`_delete_memory(memory_id=映射回真实 UUID)`
- **NONE**：一般 NOOP；但如果 metadata 中提供了 `agent_id` 或 `run_id`，会把这些会话标识补写到旧记忆 payload 中，并更新 `updated_at`

这一步体现 Mem0 的“记忆是可维护资产”的架构取向：新增并不等同于追加，而是允许在事实层面做更新与删除，并记录在 history 里。

## 6. 底层存储结构：向量库（PGVector）与 history（SQLite）

### 6.1 PGVector 的表结构：一张 collection 表承载向量与 payload

`mem0/vector_stores/pgvector.py::create_col` 会创建：

- `id UUID PRIMARY KEY`
- `vector vector(embedding_dims)`
- `payload JSONB`

可选索引：

- `diskann`（依赖 vectorscale 扩展且 dims < 2000）
- 或 `hnsw`（`vector_cosine_ops`）

### 6.2 payload 的组织方式：Mem0 的“数据面”

在 Mem0 的向量库抽象里，payload 是一个字典（PGVector 中最终作为 JSONB 写入）。以 `mem0/memory/main.py::_create_memory` 与 `_update_memory` 为准：

- **核心字段（Mem0 自己写入）**：
  - `data`：记忆文本（也就是业务侧最终看到的 `MemoryItem.memory`）
  - `hash`：`md5(data)`，用于内容指纹（写入时生成）
  - `created_at`：创建时间（字符串）
  - `updated_at`：仅在 update 或特定 NONE 分支补写会话字段时写入
- **会话隔离字段（由 `_build_filters_and_metadata` 传入并写入）**：
  - `user_id` / `agent_id` / `run_id`
  - 这些字段之所以放在 payload，而不是单独列，是为了让不同向量库实现以统一方式完成过滤。
- **参与者/消息相关字段（按路径写入）**：
  - `role`：在 `infer=False` 的逐条消息写入场景，会补充 `role`
  - `actor_id`：当消息 dict 含 `name` 时写入（同上）
- **额外业务元数据（用户传入）**：
  - `metadata` 参数会被并入 payload（作为顶层键值对的一部分），并在 `search/get` 的结果中被拆分为 `metadata` 字段（排除 core + promoted keys 后的剩余项）

从架构上看，payload 同时承担三类信息：**内容本体（data）**、**隔离维度（user/agent/run）**、**补充属性（role/actor/业务 meta）**。

### 6.3 PGVector 的写入与检索：SQL 形态如何体现会话隔离

#### 6.3.1 插入（insert）

`PGVector.insert(vectors, payloads, ids)` 会把 payload JSON 序列化，并执行：

- `INSERT INTO {collection} (id, vector, payload) VALUES (%s, %s, %s)`

这里并没有“更新已有 id 的 upsert”逻辑：Mem0 的 `_create_memory` 负责生成全新 UUID；更新走 `update`。

#### 6.3.2 检索（search）

`PGVector.search(query, vectors, limit, filters)` 过滤条件构造方式是：

- 对 `filters` 的每个 `k,v` 生成：`payload->>%s = %s`
- 多条件用 `AND` 连接，形成 `WHERE ...`

最终执行：

- `SELECT id, vector <=> %s::vector AS distance, payload FROM {collection} {WHERE...} ORDER BY distance LIMIT %s`

因此在 PGVector 场景下：

- **会话隔离**最终体现为 SQL 的 `payload->>'user_id' = '...'`（以及可选的 agent_id/run_id）
- `score` 是 `distance`（L2 距离），越小越相似

#### 6.3.3 更新/删除（update/delete/get/list）

PGVector 提供：

- `update(vector_id, vector?, payload?)`：分别更新 vector 或 payload（payload 用 Json 适配 psycopg2/3）
- `delete(vector_id)`：按 id 删除
- `get(vector_id)`：取回 payload
- `list(filters, limit)`：同样用 `payload->>` 过滤并返回 payload

### 6.4 history（SQLite）：Mem0 的“控制面审计”

Mem0 在 `Memory.__init__` 中初始化 `self.db = SQLiteManager(self.config.history_db_path)`，用于记录每条记忆的变更历史。

`mem0/memory/storage.py` 定义的 history 表 schema（当前代码）为：

- `id`（主键）
- `memory_id`（对应向量库的 UUID）
- `old_memory` / `new_memory`
- `event`（ADD/UPDATE/DELETE）
- `created_at` / `updated_at`
- `is_deleted`（逻辑删除标识）
- `actor_id` / `role`

对应到 Mem0 的写入点：

- `_create_memory`：`db.add_history(..., old=None, new=data, event="ADD", created_at=payload["created_at"], actor_id, role)`
- `_update_memory`：`db.add_history(..., old=prev_value, new=data, event="UPDATE", created_at, updated_at, actor_id, role)`
- `_delete_memory`：`db.add_history(..., old=prev_value, new=None, event="DELETE", is_deleted=1, actor_id, role)`

> 架构含义：向量库提供“检索与存储当前态”，history 记录“变更事件流”。这使得 Mem0 能提供 `history(memory_id)` 这类审计能力，而不依赖向量库本身的版本管理。

## 7. 端到端视角：客服场景的一次完整读写闭环

把业务侧（客服）与 Mem0 内部串起来，一轮典型闭环可以抽象为：

1. **检索（读路径）**：
   - `retrieve_context(query,user_id)` 调用 `Memory.search`
   - Mem0 使用 `user_id` 构造 `effective_filters`
   - 向量库按 `payload->>'user_id'` 过滤后做近邻检索，返回 `results[]`
   - 业务把 `results[].memory` 串为“相关历史信息”，喂给 LLM
2. **生成（业务侧）**：
   - 基于“当前问题 + 历史记忆”生成更一致、更个性化的回复
3. **写入（写路径）**：
   - `save_interaction(...)` 调用 `Memory.add(messages,user_id)`
   - Mem0（默认 infer=True）用 LLM 抽取 facts
   - 对每条 fact 先向量检索候选旧记忆（同一 user_id 范围内）
   - 再用 LLM 决策对候选集合做 ADD/UPDATE/DELETE/NONE
   - 执行向量库写入/更新/删除，并写 history

如果启用了图存储，读写都会额外并行产生/查询 `relations`，但不会改变业务侧 `results` 的主使用方式。

## 8. 架构要点总结（面向改造/扩展）

### 8.1 为什么 Mem0 的 add 需要“两次 LLM”

从实现看，`infer=True` 下至少包含两次 LLM 交互：

- **第一次**：事实抽取（得到 `facts`）
- **第二次**：动作决策（对候选旧记忆做 ADD/UPDATE/DELETE/NONE）

这样做的结构性收益是：

- 把“对话”转为“事实”，降低噪音与冗余
- 通过候选收敛，让第二次 LLM 决策在小集合上完成，从而可控且高效

### 8.2 数据模型的关键约束：payload 是契约中心

在当前实现里，**payload 是连接上层策略与底层存储的核心契约**：

- `search` 的过滤条件与结果字段提升（promoted keys）都依赖 payload 的键
- PGVector 通过 `payload->>key` 实现过滤
- history 通过 `memory_id` 与 `event` 记录变更，但内容仍以 payload["data"] 为中心

这意味着：

- 如果你要引入新的隔离维度（例如 tenant_id、project_id），最小改动路径是：把它纳入 `_build_filters_and_metadata` 的 metadata+filters，并确保写入 payload 与检索过滤一致。

### 8.3（实现观察）score 语义在不同向量库之间可能不一致

在 PGVector 中：

- SQL 使用 `<=>` 得到 `distance`
- `OutputData.score = distance`

但 Mem0 上层对 `threshold` 的过滤条件是 `mem.score >= threshold`。因此：

- **是否符合“越相似越保留”**，取决于 `score` 在当前 vector store 的定义（距离/相似度/倒数等）。

如果要在业务侧使用 `threshold`，建议以当前选用的 vector store 实现为准，明确其 score 语义后再设置阈值。
