# Neo4j 关键应用场景与 CRUD 进阶学习清单

你已经掌握了基础查询与写入，下一阶段建议从“会写语句”升级到“会建模、会治理、会调优、会落地业务”。

本文给你一份实战导向的学习清单：每个知识点都包含**为什么重要**、**要掌握什么**、**最小练习建议**。

---

## 1. 图建模能力（比语法更重要）

### 为什么重要
- 80% 的后续性能和可维护性问题，根因都在建模阶段。

### 要掌握什么
- 何时用“节点属性” vs “关系属性”。
- 关系方向如何定义（业务语义一致，不乱用无向）。
- 实体粒度：避免同义节点泛滥（如“北京”“北京市”）。
- 事件建模：动作是否需要独立事件节点（时间线分析常需要）。

### 最小练习
- 选一个业务场景（如“用户-订单-商品-评价”），画出节点/边 schema，并列出 5 条核心查询。

---

## 2. 约束与索引（生产必备）

### 为什么重要
- 没有约束，数据容易重复；没有索引，查询会变慢。

### 要掌握什么
- 唯一约束（防重复）：
```cypher
CREATE CONSTRAINT user_uid_unique IF NOT EXISTS
FOR (u:User) REQUIRE u.uid IS UNIQUE;
```
- 必填约束（关键属性不能为空）：
```cypher
CREATE CONSTRAINT user_uid_exists IF NOT EXISTS
FOR (u:User) REQUIRE u.uid IS NOT NULL;
```
- 普通索引（加速过滤）：
```cypher
CREATE INDEX user_name_idx IF NOT EXISTS
FOR (u:User) ON (u.name);
```

### 最小练习
- 给你自己造的一套演示模型加 1 个唯一约束 + 1 个索引，并比较查询耗时变化。

---

## 3. 完整 CRUD 技能树（不仅是 MATCH / MERGE）

### C：Create
- `CREATE`：明确新建，可能重复。
- `MERGE`：有则复用、无则创建（幂等）。

### R：Read
- 条件过滤、排序、分页、聚合、路径查询。
- `OPTIONAL MATCH` 处理“可能没有关系”的场景。

### U：Update
- `SET` 更新属性。
- `REMOVE` 删除属性或标签。
- 批量更新时注意作用域和过滤条件。

### D：Delete
- `DELETE r` 删除关系。
- `DETACH DELETE n` 删除节点及其关系。
- 分批删除（避免一次删太多导致事务压力大）。

### 最小练习
- 针对同一批数据分别实现：新增、去重新增、查询、更新、删关系、删节点。

---

## 4. 路径查询与图算法思维（图数据库核心价值）

### 为什么重要
- 图数据库优势不在“单表查字段”，而在“关系链路推理”。

### 要掌握什么
- 可变长度路径：
```cypher
MATCH p = (a:User {uid:'u1'})-[:KNOWS*1..3]->(b)
RETURN p LIMIT 20;
```
- 最短路径：
```cypher
MATCH p = shortestPath((a:User {uid:'u1'})-[:KNOWS*..10]-(b:User {uid:'u9'}))
RETURN p;
```
- 路径过滤（关系类型、节点属性条件）。

### 最小练习
- 实现“1~3 跳好友推荐”与“两个用户最短关系链”。

---

## 5. 批量写入与批处理（数据量上来后必须会）

### 为什么重要
- 单条写入在大数据量下太慢；大事务又会占内存。

### 要掌握什么
- `UNWIND` 批量参数写入：
```cypher
UNWIND $rows AS row
MERGE (u:User {uid: row.uid})
SET u.name = row.name, u.updated_at = timestamp();
```
- 分批提交（应用层分块，常见 500~5000 一批，按实际压测定）。

### 最小练习
- 用一组 1k~5k 的模拟数据，比较“逐条写入”与“UNWIND 批写”速度差异。

---

## 6. 查询性能优化（会看 PROFILE / EXPLAIN）

### 为什么重要
- 业务变慢时，第一技能不是“猜”，而是看执行计划。

### 要掌握什么
- `EXPLAIN` 看计划，不执行。
- `PROFILE` 看真实执行、行数、DB hits。
- 避免早期笛卡尔积（不相关 MATCH 并列）。
- 尽早过滤（把 `WHERE` 往前放）。

### 最小练习
- 对同一个查询做两版写法，用 `PROFILE` 对比 DB hits 和耗时。

---

## 7. 事务与并发一致性

### 为什么重要
- 并发写入时容易出现重复、覆盖、竞争更新。

### 要掌握什么
- 用 `MERGE + 约束` 保证幂等写入。
- 理解事务边界（应用层一次业务操作尽量在同一事务里）。
- 失败重试策略（尤其网络抖动、瞬时冲突）。

### 最小练习
- 并发模拟同一实体写入，验证最终是否只保留一份节点。

---

## 8. 数据质量治理（图会“长脏”，要会清洗）

### 为什么重要
- 图库一旦积累脏数据（同义、错边、孤儿点），查询质量会持续下降。

### 要掌握什么
- 重复实体识别与合并策略（规则 + 相似度）。
- 错误关系回滚/删除流程。
- 质量巡检报表：孤立节点数、低频关系类型、异常高出度节点。

### 最小练习
- 写 3 条巡检查询：孤立点、重复名节点、异常高出度节点。

---

## 9. 安全与多租户隔离（你当前 mem0 场景非常关键）

### 为什么重要
- 你的 `graph_memory.py` 已用 `user_id/agent_id/run_id` 做隔离，不统一会串数据。

### 要掌握什么
- 所有查询都显式带租户过滤条件。
- 运维脚本避免“全库误删”（慎用无过滤 `DETACH DELETE`）。
- 参数化查询，避免拼接注入风险。

### 最小练习
- 随机抽 10 条查询，检查是否都带了租户过滤字段。

---

## 10. 与向量检索结合（你当前项目的核心特色）

### 为什么重要
- 你不是纯图应用，而是“向量 + 图”双路检索。

### 要掌握什么
- 什么时候用向量召回，什么时候用图关系补充解释。
- 相似度阈值如何影响“节点合并 vs 新建”。
- 图关系如何用于 rerank 或输出可解释证据。

### 最小练习
- 设计一个问答场景：先向量召回，再附加 1 跳关系作为解释信息。

## 12. 自检清单（达到这些才算进阶）

- 能独立设计一个业务子图模型（节点/关系/属性/方向）。
- 能写出带索引与约束的初始化脚本。
- 能解释 `CREATE`、`MERGE`、`ON CREATE`、`ON MATCH` 的差别与适用场景。
- 能用 `PROFILE` 找出慢查询并改写。
- 能做批量导入，不靠逐条插入。
- 能建立基础数据治理规则，避免图逐步失真。
- 能把图关系真正用于业务解释或决策，而不是只做展示。
