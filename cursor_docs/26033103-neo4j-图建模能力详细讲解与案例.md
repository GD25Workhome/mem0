## 1. 什么是“图建模能力”

一句话：**把业务世界抽象成“稳定节点 + 明确关系 + 可演进属性”的能力**。

图建模不是先写语句，而是先回答三件事：
1. 什么东西应该成为节点（Entity）？
2. 两个节点之间到底是什么关系（Relation）？
3. 这个信息应该放在节点属性，还是关系属性？

如果这三件事做得好，后面的查询、性能、扩展都会轻松很多。

---

## 2. 一套实战建模方法（建议每次按这个顺序）

### 第一步：先列“业务问题”，再建图

先写你要回答的问题，例如电商场景：
1. 某用户最近买了什么？
2. 某商品被哪些用户高分评价？
3. 某订单包含哪些商品？
4. 某用户是否复购某类商品？
5. 某商品常和哪些商品一起被购买？

> 原则：**查询问题决定模型，不是模型反过来硬凑查询**。

### 第二步：识别核心实体（节点）

常见实体节点：
- `User`
- `Order`
- `Product`
- `Review`

### 第三步：识别关系（边）和方向

建议关系语义：
- `(u:User)-[:PLACED]->(o:Order)` 用户下单
- `(o:Order)-[:CONTAINS]->(p:Product)` 订单包含商品
- `(u:User)-[:WROTE]->(r:Review)` 用户写评价
- `(r:Review)-[:REVIEWS]->(p:Product)` 评价对应商品

方向规则：
- 使用**可读成动词短语**的方向（从主语到宾语）
- 避免同一语义同时出现双方向（后期很难治理）

### 第四步：决定属性放节点还是关系

这是最容易踩坑的点，推荐规则：

1. **描述实体本身的稳定特征** -> 放节点属性  
   - `User.name`、`Product.category`
2. **描述两个实体之间这次连接的事实** -> 放关系属性  
   - `[:PLACED {at, channel}]`、`[:CONTAINS {qty, price_at_order}]`
3. **会被多次复用/独立生命周期的信息** -> 考虑独立节点  
   - 评价常需要审核、修改、追溯，适合独立 `Review` 节点

### 第五步：设计唯一标识与约束

例如：
- `User.uid` 唯一
- `Order.oid` 唯一
- `Product.pid` 唯一
- `Review.rid` 唯一

这样可以配合 `MERGE` 做幂等写入，避免重复数据。

## 3. 你提到的关系图（用户-订单-商品-评价）：我的理解

你在原文里给了最小练习：  
“选一个业务场景（如 用户-订单-商品-评价），画出节点/边 schema，并列出 5 条核心查询。”

我对这个关系图的理解是：它至少有两层价值。

### 第一层：交易链路（Who bought What）

核心路径：
`User -> Order -> Product`

能回答：
- 用户买了哪些商品
- 商品被哪些用户买过
- 订单内商品组合

### 第二层：反馈链路（Who said What about What）

核心路径：
`User -> Review -> Product`

能回答：
- 用户对某商品的评分历史
- 某商品的评价结构（星级、情感）
- 评价与购买行为是否一致（买了但给低分）

### 这两层为什么要分开建模

因为“购买事实”和“评价事实”是两类不同事件：
- 购买是订单系统主事实（金额、数量、渠道）
- 评价是内容系统主事实（评分、文本、时间）

把两者混成一条关系会导致：
- 属性爆炸（字段越来越杂）
- 语义混乱（同一关系既像交易又像反馈）
- 查询复杂（很多条件要靠 if 拼）

所以建议拆成 `Order` 与 `Review` 两类事件节点，既清晰又可扩展。

## 4. 一个可直接落地的 schema 建议

### 节点标签与关键属性

- `(:User {uid, name, city, created_at})`
- `(:Order {oid, status, total_amount, created_at})`
- `(:Product {pid, title, category, brand})`
- `(:Review {rid, rating, content, created_at})`

### 关系与关系属性

- `(:User)-[:PLACED {at, channel}]->(:Order)`
- `(:Order)-[:CONTAINS {qty, price_at_order}]->(:Product)`
- `(:User)-[:WROTE {at}]->(:Review)`
- `(:Review)-[:REVIEWS]->(:Product)`

### 为什么这样拆

1. `Order` 与 `Review` 都是“事件节点”，天然可追溯
2. 订单内商品的数量/成交价属于“订单-商品关系事实”，放在 `CONTAINS` 上更合理
3. 评价文本和评分是评价本体，放在 `Review` 节点更便于文本分析和审核流

## 5. 面向查询的建模验证（5 条核心查询）

下面 5 条是“建模是否合理”的验收题。  
如果这 5 条都能自然写出来，说明模型方向基本正确。

### 查询 1：某用户最近 30 天买了哪些商品

```cypher
MATCH (u:User {uid: $uid})-[:PLACED]->(o:Order)-[:CONTAINS]->(p:Product)
WHERE o.created_at >= datetime() - duration('P30D')
RETURN p.pid, p.title, count(*) AS times
ORDER BY times DESC;
```

### 查询 2：某商品近 90 天平均评分

```cypher
MATCH (:Product {pid: $pid})<-[:REVIEWS]-(r:Review)
WHERE r.created_at >= datetime() - duration('P90D')
RETURN avg(r.rating) AS avg_rating, count(r) AS review_count;
```

### 查询 3：买过某商品且给过低分（<=2）的用户

```cypher
MATCH (u:User)-[:PLACED]->(:Order)-[:CONTAINS]->(p:Product {pid: $pid})
MATCH (u)-[:WROTE]->(r:Review)-[:REVIEWS]->(p)
WHERE r.rating <= 2
RETURN DISTINCT u.uid, u.name, r.rating, r.content;
```

### 查询 4：商品共购（与目标商品同单出现）

```cypher
MATCH (:Product {pid: $pid})<-[:CONTAINS]-(o:Order)-[:CONTAINS]->(other:Product)
WHERE other.pid <> $pid
RETURN other.pid, other.title, count(*) AS co_buy
ORDER BY co_buy DESC
LIMIT 20;
```

### 查询 5：某用户“买了但没评价”的商品

```cypher
MATCH (u:User {uid: $uid})-[:PLACED]->(:Order)-[:CONTAINS]->(p:Product)
WHERE NOT EXISTS {
  MATCH (u)-[:WROTE]->(:Review)-[:REVIEWS]->(p)
}
RETURN DISTINCT p.pid, p.title;
```

## 6. 4 个常见建模错误（你要主动规避）

### 错误 1：把所有信息都塞进节点属性

后果：关系语义消失，图退化成“多张表拼一起”。

### 错误 2：关系类型过于泛化（全叫 `RELATED`）

后果：查询可读性差，业务含义不清晰。

### 错误 3：同一语义关系方向不统一

后果：写查询要双向兼容，性能和维护都差。

### 错误 4：没有唯一键与约束

后果：`MERGE` 也会出现重复实体（键定义不完整时尤其明显）。

## 9. 后续学习待办：图数据库 vs 关系型数据库（未完成）

这一节用于承接当前阶段的疑问：  
“`用户-订单-商品-评价` 这个案例里，图数据库是否真的比关系型数据库有优势？”

### 9.1 当前结论（阶段性）

1. 你的直觉是合理的：这个案例中的很多基础查询，关系型数据库同样可以很好完成。  
2. 图数据库并不是“通用替代关系库”，而是在特定问题类型上优势明显。  
3. 很多生产系统会采用组合方案：  
   - 关系库负责主交易与强事务  
   - 图库负责关系分析、路径推理、推荐/风控解释层

### 9.2 什么时候关系库更合适

- 查询主要是固定结构、单跳或少量 join
- 以 OLTP 事务写入、强一致、报表统计为主
- 数据关系模式长期稳定，变化不大

### 9.3 什么时候图库更合适

- 需要多跳关系探索（跳数不固定）
- 需要路径语义与路径解释（why this result）
- 关系密集型问题（推荐、反欺诈、团伙识别、传播链分析）
- 查询逻辑更像“找关系链路”而不是“查一行记录”

### 9.4 判断清单（自测）

如果下面问题多数回答“是”，可以优先考虑图数据库（或引入图库作为补充层）：

- [ ] 业务查询是否经常需要 2 跳以上关系推理？  
- [ ] 查询跳数是否不固定（如 1~N 跳）？  
- [ ] 是否需要输出可解释路径，而不只是结果值？  
- [ ] 是否存在网络结构问题（推荐、风控、传播）？  
- [ ] 关系模式是否经常演进（新增关系类型频繁）？
