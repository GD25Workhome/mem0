# Mem0 search 实现逻辑与关键技术细节

本文档梳理从业务侧调用 `mem0.search(query, user_id=user_id)` 到向量库返回结果的完整实现链路，并说明关键技术点。涉及文件：`GD25/express_customer_service copy.py`、`mem0/memory/main.py`、`mem0/vector_stores/pgvector.py`。

---

## 1. 调用入口（业务侧）

**文件**：`GD25/express_customer_service copy.py`

```python
def retrieve_context(self, query: str, user_id: str) -> List[Dict]:
    # ...
    memories = self.mem0.search(query, user_id=user_id)
    # memories 结构: {"results": [{"id", "memory", "score", ...}, ...]}
```

- **含义**：按当前用户 `user_id` 在 Mem0 中检索与 `query` 相关的记忆，用于拼进 LLM 上下文。
- **约束**：`search` 要求至少传入 `user_id`、`agent_id`、`run_id` 之一，否则会报错。

---

## 2. Memory.search 主流程（mem0/memory/main.py）

**入口**：`Memory.search(self, query, *, user_id=None, agent_id=None, run_id=None, limit=100, filters=None, threshold=None, rerank=True)`

### 2.1 构建查询过滤器（effective_filters）

```python
_, effective_filters = _build_filters_and_metadata(
    user_id=user_id, agent_id=agent_id, run_id=run_id, input_filters=filters
)
```

- **`_build_filters_and_metadata`**（约 87–163 行）：
  - 根据传入的 `user_id` / `agent_id` / `run_id` 生成**存储用 metadata 模板**和**查询用 effective_filters**。
  - 至少需要一种会话维度，否则抛出 `Mem0ValidationError`。
  - 会把 `user_id` 等写入 `effective_query_filters`，例如：`effective_filters["user_id"] = user_id`。
- 若传入了 `filters` 且包含高级操作符（如 `eq`、`in`、`gt` 等），会通过 `_process_metadata_filters` 转成向量库可用的条件并合并进 `effective_filters`；否则简单 merge。

### 2.2 校验与遥测

- 再次校验 `effective_filters` 中至少包含 `user_id`、`agent_id`、`run_id` 之一。
- 调用 `process_telemetry_filters`、`capture_event("mem0.search", ...)` 做遥测（不改变检索逻辑）。

### 2.3 并发执行：向量检索 + 图检索（可选）

```python
with concurrent.futures.ThreadPoolExecutor() as executor:
    future_memories = executor.submit(
        self._search_vector_store, query, effective_filters, limit, threshold
    )
    future_graph_entities = (
        executor.submit(self.graph.search, query, effective_filters, limit)
        if self.enable_graph else None
    )
    concurrent.futures.wait(...)
    original_memories = future_memories.result()
    graph_entities = future_graph_entities.result() if future_graph_entities else None
```

- **向量检索**：`_search_vector_store(query, effective_filters, limit, threshold)` 是主路径，返回 `original_memories`。
- **图检索**：仅当 `enable_graph=True` 时在另一线程执行，结果放在 `graph_entities`，最终返回结构为 `{"results": ..., "relations": graph_entities}`。

### 2.4 重排序（Rerank）

- 若 `rerank=True` 且配置了 `self.reranker` 且 `original_memories` 非空，则调用 `self.reranker.rerank(query, original_memories, limit)`，用结果覆盖 `original_memories`；失败时仅打日志，保留原结果。

### 2.5 返回结构

- 无图：`{"results": original_memories}`。
- 有图：`{"results": original_memories, "relations": graph_entities}`。
- 业务侧主要使用 `memories["results"]`，每条为 `MemoryItem` 的字典形式（含 `id`、`memory`、`score` 等）。

## 3. _search_vector_store：向量检索核心（mem0/memory/main.py）

**方法**：`_search_vector_store(self, query, filters, limit, threshold=None)`（约 954–992 行）

### 3.1 查询向量化

```python
embeddings = self.embedding_model.embed(query, "search")
```

- 使用配置的 `embedding_model`（如 OpenAI 兼容的 text-embedding-v1）对 `query` 做一次向量化。
- 第二参数 `"search"` 为 `memory_action`，部分 embedder 可根据 `add`/`search`/`update` 使用不同策略，当前多数实现中与 `add` 一致。

### 3.2 调用向量库

```python
memories = self.vector_store.search(
    query=query, vectors=embeddings, limit=limit, filters=filters
)
```

- 传入的是**已算好的向量** `embeddings`，以及**过滤条件** `filters`（即上文的 `effective_filters`，含 `user_id` 等）。
- 向量库返回的是“原始”结果列表，元素通常包含 `id`、`payload`、`score`（或距离，见下节）。

### 3.3 结果规范化与 threshold 过滤

- 遍历 `memories`，将每条转为统一的 `MemoryItem` 字典：
  - `memory` 来自 `payload["data"]`，`score` 来自向量库返回的 score/distance。
  - 提升字段如 `user_id`、`agent_id`、`run_id`、`actor_id`、`role` 到顶层，其余放入 `metadata`。
- **threshold**：若 `threshold is not None`，仅保留 `mem.score >= threshold` 的项；否则全部保留。
- **注意**：在 PGVector 中，`score` 实际为**距离**（见下），即“越小越相似”；若使用 `threshold`，需理解当前语义是“最小 score”，与“相似度越大越好”的直觉可能相反，具体以各 vector store 实现为准。

## 4. PGVector 的 search 实现（mem0/vector_stores/pgvector.py）

**方法**：`PGVector.search(self, query, vectors, limit=5, filters=None)`（约 202–244 行）

### 4.1 过滤条件构造

```python
filter_conditions = []
filter_params = []
if filters:
    for k, v in filters.items():
        filter_conditions.append("payload->>%s = %s")
        filter_params.extend([k, str(v)])
filter_clause = "WHERE " + " AND ".join(filter_conditions) if filter_conditions else ""
```

- `effective_filters` 中的键值对（如 `user_id=xxx`）被转成 SQL 的 `payload->>'user_id' = 'xxx'` 形式。
- 多个条件用 `AND` 连接，实现**按 user_id（及可选 agent_id/run_id 等）过滤**，只在该用户的记忆集合内做相似度检索。

### 4.2 向量相似度查询

```python
cur.execute(
    f"""
    SELECT id, vector <=> %s::vector AS distance, payload
    FROM {self.collection_name}
    {filter_clause}
    ORDER BY distance
    LIMIT %s
    """,
    (vectors, *filter_params, limit),
)
```

- **`<=>`**：pgvector 的 L2 距离算子，**值越小表示越相似**。
- 在**已过滤**的表上按 `distance` 排序，取前 `limit` 条。
- 返回的 `distance` 被放入 `OutputData.score`，因此在 PGVector 中 **score 表示的是距离**，不是 0~1 的相似度。

### 4.3 返回结构

- 每条结果为 `OutputData(id, score, payload)`，与 `_search_vector_store` 中期望的 `mem.id`、`mem.score`、`mem.payload` 一致。

## 5. 端到端数据流小结

| 阶段 | 位置 | 输入 | 输出/作用 |
|------|------|------|-----------|
| 业务调用 | express_customer_service | `query`, `user_id` | 得到 `memories["results"]` |
| 构建过滤 | main.py `_build_filters_and_metadata` | `user_id` 等 | `effective_filters`（含 user_id） |
| 向量检索 | main.py `_search_vector_store` | query, effective_filters, limit, threshold | 原始记忆列表 |
| 查询向量化 | main.py → embedder | query | embeddings |
| 向量库检索 | pgvector.search | vectors, filters, limit | id, distance(score), payload |
| SQL 过滤 | pgvector | filters → payload->>key = value | 仅该 user 的记忆 |
| 规范化 + threshold | _search_vector_store | 原始 memories, threshold | MemoryItem 列表 |
| 可选 Rerank | main.py search | query, original_memories, limit | 重排后的 results |
| 返回 | main.py search | — | `{"results": [...], "relations": ...?}` |

---

## 6. 关键技术点总结

1. **会话隔离**：通过必须传入的 `user_id`（或 agent_id/run_id）在 `_build_filters_and_metadata` 中生成 `effective_filters`，在 PGVector 中转为 `payload->>'user_id' = ...`，保证检索只在该用户（或 agent/run）的记忆范围内进行。
2. **检索路径**：query → embed(query) → vector_store.search(vectors, filters, limit) → 规范化 + threshold → 可选 rerank → results。
3. **PGVector 语义**：使用 L2 距离 `<=>`，返回的 `score` 为距离（越小越相似）；若使用 `threshold`，需注意其“最小 score”在当前实现下对应的是“最大允许距离”的语义（视具体 store 而定）。
4. **可扩展参数**：`limit`、`filters`（含高级操作符）、`threshold`、`rerank` 均在 `Memory.search` 中支持，业务侧可按需传入（如 `limit`、`threshold`、`rerank=False` 等）。

以上即为从 `express_customer_service copy.py` 中 `mem0.search(query, user_id=user_id)` 到 PGVector 的完整实现逻辑与关键技术细节。
