# Neo4j 数据质量治理：巡检与清洗实战指南

对应总览章节：[`26033102-neo4j-关键应用场景与CRUD进阶学习清单.md`](26033102-neo4j-关键应用场景与CRUD进阶学习清单.md) 的 `## 8. 数据质量治理（图会“长脏”，要会清洗）`。

目标：通过一套“可直接在你库中跑”的巡检与清洗示例，帮助你建立图数据质量治理的基本意识和工具箱。

---

## 1. 为什么图容易“长脏”

典型脏数据类型：
- **重复实体**：同一个真实对象被建成多个节点（北京/北京市、uid 大小写不同等）
- **错误关系**：边连错对象或方向错误（用户指向错误订单、类型误标）
- **孤立节点**：没有任何关系的节点（历史残留、导入失败的半成品）
- **异常结构**：某些节点出度/入度异常高，可能是导入脚本 Bug 或业务异常

图一旦变脏，问题会逐步放大：
- 查询结果不可信（推荐、分析、解释全受影响）
- 清理成本随时间指数级上升

---

## 2. 巡检思路总览（先有大图，再看细节）

你可以把图数据质量巡检拆成三类查询：
1. **结构巡检**：看整体结构有没有明显异常（孤立点、高出度节点等）
2. **内容巡检**：看属性层面的重复与异常（重复名、非法值）
3. **关系巡检**：看边的类型分布与异常（低频类型、可疑方向）

建议先在**测试或只读环境**跑这些查询，再决定是否在生产做定期巡检。

下面给出“最小练习”要求的 3 条巡检 + 一些扩展示例。

---

## 3. 巡检 1：孤立节点（没有任何边的节点）

### 3.1 针对某个标签检查孤立节点

以 `User` 为例：

```cypher
// 找出没有任何关系的 User 节点
MATCH (u:User)
WHERE NOT (u)--()
RETURN u.uid AS uid, u.name AS name
LIMIT 50;
```

解释：
- `NOT (u)--()` 表示该节点没有任何入边或出边
- 这类节点可能是：导入残留、事务失败一半留下的“壳”

### 3.2 统计孤立节点占比

```cypher
MATCH (u:User)
WITH count(u) AS total
MATCH (u:User)
WHERE NOT (u)--()
RETURN count(u) AS isolated, total, toFloat(count(u)) / total AS ratio;
```

---

## 4. 巡检 2：重复名节点（潜在重复实体）

> 注意：**“重复名”不等于“必然重复实体”**，但常是合并候选。

### 4.1 按名字分组找重复

```cypher
// 找出 name 重复出现的 User
MATCH (u:User)
WHERE exists(u.name)
WITH u.name AS name, collect(u) AS users, count(*) AS cnt
WHERE cnt > 1
RETURN name, cnt, [u IN users | u.uid][0..10] AS sample_uids
ORDER BY cnt DESC
LIMIT 50;
```

解释：
- `cnt > 1` 表示同名出现多次
- `sample_uids` 给你部分 uid，方便人工核查

### 4.2 对接你的 mem0 图（参考）

如果你的实体标签是 `__Entity__` 或其他图记忆节点，可以把 `:User` 改成对应标签，并按 `name + user_id` 分组，避免跨用户串数据。

---

## 5. 巡检 3：异常高出度节点（可能的数据/业务异常）

> 出度高并不一定是错，但往往值得关注：  
> 例如一个用户指向几万条订单、一个 IP 关联大量账号等。

### 5.1 统计出度最高的若干节点

以 `User` 发出的 `PLACED` 关系为例：

```cypher
MATCH (u:User)-[r:PLACED]->()
WITH u, count(r) AS out_degree
RETURN u.uid AS uid, u.name AS name, out_degree
ORDER BY out_degree DESC
LIMIT 50;
```

### 5.2 针对所有关系的出度汇总

```cypher
MATCH (n)-[r]->()
WITH n, count(r) AS out_degree
RETURN labels(n) AS labels, n.uid AS uid, out_degree
ORDER BY out_degree DESC
LIMIT 50;
```

观察点：
- 如果某类节点理论上出度应该很小，但这里异常偏大，可能是：
  - 导入脚本重复写入同一边
  - 业务 Bug 创建了错误关系

## 6. 额外巡检示例（可按需扩展）

### 6.1 低频关系类型（“冷门边”）

```cypher
MATCH ()-[r]->()
WITH type(r) AS rel_type, count(*) AS cnt
RETURN rel_type, cnt
ORDER BY cnt ASC
LIMIT 20;
```

用途：
- 看有没有几乎不被使用的关系类型（可能是历史遗留、拼写错误）

### 6.2 边属性异常值

以 `rating` 为例，评分应该在 1~5：

```cypher
MATCH (:User)-[r:RATED]->(:Product)
WHERE r.rating < 1 OR r.rating > 5
RETURN r.rating AS bad_rating, count(*) AS cnt
ORDER BY cnt DESC;
```

## 7. 清洗策略入门（只给思路，改动前要小心）

> 生产上的清洗要非常谨慎，建议先在测试或备份数据上演练。

### 7.1 合并重复实体（高风险操作，先设计规则）

简单示意（不要直接在线上执行）：

```cypher
// 假设你已经人工确认 name 相同的一组 uid 应合并
MATCH (u1:User {uid: $keep_uid})
MATCH (u2:User {uid: $remove_uid})

// 把指向 u2 的关系迁移到 u1
MATCH (u2)-[r]->(x)
MERGE (u1)-[r_new:TYPE(r)]->(x)
SET r_new += r
DELETE r;

MATCH (y)-[r2]->(u2)
MERGE (y)-[r2_new:TYPE(r2)]->(u1)
SET r2_new += r2
DELETE r2;

// 删除被合并节点
DETACH DELETE u2;
```

> 实际使用时，需要更严格的规则与回滚方案，这里仅作为“概念示例”，提醒你合并应包括“关系迁移 + 节点删除”两步。

### 7.2 删除明显脏数据（例如某类非法节点）

```cypher
// 示例：删除 name 为空且没有任何关系的 User
MATCH (u:User)
WHERE (u.name IS NULL OR trim(u.name) = '') AND NOT (u)--()
DETACH DELETE u;
```

## 8. 最小练习方案（建议你在自己库中做）

> 强烈建议在**测试环境或非关键标签**上尝试，避免直接动线上核心数据。

### 练习步骤

1. 挑一个标签作为练习对象（例如 `User` 或你 mem0 图里的某个实体标签）  
2. 写并执行：
   - 孤立节点巡检查询  
   - 重复名节点巡检查询  
   - 高出度节点巡检查询  
3. 记录每条查询的前 20 行结果，人工判断哪些是真问题、哪些是业务合理现象  
4. 从中挑出 1~2 条“可安全修复”的脏数据，设计一条清洗 Cypher（先在小样本/测试数据上验证）  
5. 在笔记中记录：  
   - 你采用的巡检规则  
   - 识别出的典型问题  
   - 清洗前后差异

## 9. 建议的生产治理节奏（给未来参考）

- **日/周级巡检**：跑一遍结构与内容巡检查询，产出报表  
- **月度治理**：挑选高风险问题做专项清理（重复实体、错误关系）  
- **新特性上线前**：评估是否会引入新的脏数据模式（例如新关系类型/导入脚本）  
- **与监控结合**：将“异常高出度节点数”“孤立节点数”等指标接入监控系统
