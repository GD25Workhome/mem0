## Mem0 速览：记忆查询、记忆新增、底层存储（快速阅读版）

本文是对 `26031801-mem0-记忆检索新增与存储架构梳理.md` 的**精简版**，目标是让你在几分钟内抓住 Mem0 的关键架构与数据流：**怎么查、怎么写、存在哪**。不展开代码细节，只保留机制与边界。

---

## 1. 总体定位：Mem0 在业务里扮演什么角色

在典型对话型应用中（例如客服助手），Mem0 负责把“对话”变成可复用的“记忆”，并在下一次对话中把相关记忆找回来：

- **读路径（search）**：把与当前问题相关的历史事实/偏好检索出来，注入到提示词上下文，让回复更一致、更个性化。
- **写路径（add）**：把本轮对话提炼成“事实级记忆”，并对历史记忆做增量维护（新增/更新/删除），避免无意义堆叠。

---

## 2. 三层结构：从接口到存储的清晰分工

可以把 Mem0 理解成三层（从上到下）：

- **接口层（Facade）**：`Memory.search` / `Memory.add`
  - 统一参数与返回结构
  - 强制会话范围（user/agent/run）隔离
  - 并发编排（向量库 + 可选图存储）
- **编排层（Extraction + Policy）**
  - `search`：过滤范围 → 向量检索 →（可选）rerank → 统一结果
  - `add(infer=True)`：事实抽取 → 候选旧记忆收敛 → 动作决策 → 执行落库
- **存储层（Vector Store + History + 可选 Graph）**
  - 向量库：近邻检索 + payload 过滤（按 user/agent/run）
  - history（SQLite）：记录每条记忆的 ADD/UPDATE/DELETE 变更流
  - 图存储（可选）：实体/关系的补充通路，返回 `relations`

## 3. 记忆查询（search）速览：从问题到 results

### 3.1 一句话链路

**构造会话过滤 → 向量化 query → 向量库检索（带过滤）→（可选）重排序 → 返回 results**

### 3.2 关键输入/输出

- **输入**：
  - `query`：本轮用户问题
  - **会话范围**：`user_id` / `agent_id` / `run_id` 至少一个（否则不允许检索）
  - 可选：`limit`、`filters`、`threshold`、`rerank`
- **输出**：
  - 标准返回：`{"results": [...]}`（若启用图存储，还会有 `relations`）
  - `results` 里每条记忆的核心就是 `**memory` 文本**，业务通常只消费这一项用于拼接上下文

### 3.3 这条链路的架构意义

- **会话隔离是“第一原则”**：所有检索都严格限定在 `user/agent/run` 过滤范围内，避免跨用户串记忆。
- **结果结构稳定**：业务侧可以把 Mem0 当“记忆检索服务”，不依赖向量库实现细节。

## 4. 记忆新增（add）速览：从对话到可维护的记忆资产

### 4.1 两种写入模式

- **模式 A：`infer=False`（更像日志）**
  - 把每条消息当作一条记忆写入
  - 优点：简单直接
  - 风险：噪音多、冗余高、后续检索质量容易下降
- **模式 B：`infer=True`（默认，事实级维护）**
  - 把对话提炼成 facts，再对既有记忆做增量更新
  - 优点：记忆更“干净”、更稳定、更可复用

### 4.2 `infer=True` 的核心机制（只保留架构要点）

按阶段拆解为四步：

1. **事实抽取**：从多轮 messages 中抽出 `facts`（列表）
2. **候选收敛**：对每条 fact 先在向量库中搜索一小批“可能相关的旧记忆”（同会话范围内）
3. **动作决策**：在“新 facts + 候选旧记忆”这个小集合上，决策对每条记忆执行：
  - `ADD`：新增一条记忆
  - `UPDATE`：更新某条旧记忆的内容
  - `DELETE`：删除某条旧记忆
  - `NONE`：不做内容变更（某些情况下会补写会话字段）
4. **执行落库**：把动作落到向量库（insert/update/delete），并写入 history（形成审计轨迹）

### 4.3 这条链路的架构意义

- **写入不是“追加”，而是“维护”**：Mem0 把记忆当作长期资产，允许更新与删除，从而让记忆库随时间演进而不失控膨胀。
- **先收敛再决策**：用向量检索先把“相关旧记忆”收敛成小集合，再做策略决策，保证成本与结果可控。

## 5. 底层存储速览：记忆究竟存在哪、怎么被过滤

### 5.1 向量库：当前态 + 相似度检索

以 PGVector 为例（其余向量库同属这一抽象层）：

- **表的核心三元组**：`id` + `vector` + `payload`
- **payload 的关键作用**：
  - 存放 `data`（记忆文本）
  - 存放 `user_id/agent_id/run_id`（会话隔离维度）
  - 存放额外元数据（可选）
- **过滤方式的本质**：检索时把“会话隔离维度”作为过滤条件，只在该范围内做向量近邻搜索

### 5.2 history（SQLite）：变更流/审计

Mem0 同时维护一份本地的 history 表，用于记录：

- 每次对某条记忆的 **ADD/UPDATE/DELETE**
- 变更前后文本（old/new）
- 时间字段与可选的参与者信息（actor/role）

理解上可以类比为：

- 向量库：**当前态可检索的记忆集合**
- history：**记忆如何一步步演化的事件流**

### 5.3 图存储（可选）：补充结构化关系

如果启用图存储：

- `add` 会并行写入图实体/关系
- `search` 会并行返回 `relations`

但它通常是对 `results` 的补充通路，不改变“向量检索 + 会话隔离”作为主路径的事实。

## 6. 快速决策指南：你在业务侧该怎么用

- **最小可用集**：
  - 写：`add(messages, user_id=...)`（默认 infer=True）
  - 读：`search(query, user_id=...)`
- **当你希望“记忆更像事实库”**：保持 `infer=True`（默认）
- **当你希望“记忆更像对话日志”**：显式 `infer=False`（但要接受噪音与冗余）
- **当你需要更强的可解释/可审计**：利用 `history(memory_id)` 回看记忆演化

---

## 7. 一页纸总结

- **search**：会话隔离过滤 + 向量近邻检索 +（可选）rerank/graph → `{"results":[...]}`  
- **add**：对话 → facts → 候选收敛 → 动作决策（ADD/UPDATE/DELETE/NONE）→ 向量库落库 + history 审计  
- **存储**：向量库负责“当前态检索”，history 负责“变更流审计”，图存储（可选）负责“关系补充”
